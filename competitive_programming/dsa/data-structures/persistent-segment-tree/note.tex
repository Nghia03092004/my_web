\section*{Overview}

Persistent segment tree is the structure for ``keep every past version.'' Instead of mutating one segment tree in place,
every point update creates a new root and only clones the \(O(\log n)\) nodes on the updated path.

\section*{When to Use It}

Use it when:

\begin{itemize}
  \item queries refer to historical versions,
  \item each update touches one logarithmic root-to-leaf path,
  \item prefix versions or time-stamped versions make offline processing natural.
\end{itemize}

The most common contest uses are prefix frequency trees, k-th order statistics on subarrays, and historical range sums.

\section*{Core Idea}

If one point update only changes the nodes covering that point, then all other nodes are still valid for the new
version. So copy only the changed path and let the untouched subtrees be shared.

\section*{Key Insight}

Persistence is cheap here because immutability and segment-tree locality match perfectly. One update affects one path,
so one version needs only one path of fresh nodes.

\section*{Worked Problem}

\paragraph{Problem.}
Maintain an array under point updates. After each update, keep the new array version and answer range-sum queries on any
past version.

\paragraph{Why persistence fits.}
Version \(v_t\) differs from version \(v_{t-1}\) at one position. Therefore only one root-to-leaf path changes.

\paragraph{Version view.}
If update \(t\) changes position \(p\), then:

\begin{itemize}
  \item nodes whose segments do not contain \(p\) are reused,
  \item nodes whose segments contain \(p\) are cloned and recomputed.
\end{itemize}

\section*{Example Problem Pattern}

For k-th smallest queries on subarray \([l,r]\), build one persistent frequency tree per prefix. Then compare version
roots for prefixes \(r\) and \(l-1\). The difference between those roots acts like the multiset of the subarray.

\section*{Correctness Intuition}

Each version root points to a tree whose aggregates reflect exactly the updates that were applied before that version was
created. Shared nodes are safe because no future operation mutates them; newer versions only allocate new nodes.

\section*{Complexity Analysis}

\begin{itemize}
  \item one point update: \(O(\log n)\) time and \(O(\log n)\) new nodes,
  \item one query on a chosen version: \(O(\log n)\),
  \item memory after \(q\) updates: \(O(n + q \log n)\).
\end{itemize}

\section*{Implementation}

The sample code uses an index-based node pool and supports:

\begin{itemize}
  \item build,
  \item point update returning a new root,
  \item range-sum query on any root.
\end{itemize}

\section*{Common Pitfalls}

\begin{itemize}
  \item Mutating an old node instead of cloning it.
  \item Forgetting to store each new root.
  \item Underestimating memory usage when \(q\) is large.
  \item Reaching for persistence when simple offline prefix accumulation would already solve the problem.
\end{itemize}

\section*{Variants / Extensions}

\begin{itemize}
  \item Persistent frequency tree for order statistics.
  \item Persistent lazy trees, which exist but are much easier to get wrong.
  \item Other path-copy persistent structures such as treaps.
\end{itemize}

\section*{Practice Problems}

\begin{itemize}
  \item Historical point updates with range queries on old versions.
  \item K-th smallest value on subarrays using prefix versions.
  \item Offline queries where every prefix snapshot must remain accessible.
\end{itemize}
